## What Went Well

1. **Robust Final Polish:** Despite high review churn, the final deliverables are exceptionally thoroughly tested, covering deep edge cases like optimistic UI rollbacks and defensive DB lock handling.
2. **Deep Edge Case Discovery:** The AI code reviews successfully caught subtle but severe bugs, including race conditions, broken double-submit Enter+blur UI guards, and N+1 lazy-loading regressions.
3. **Backend Operational Stability:** Integrated standard `503 Service Unavailable` handling for transient DB connections across all endpoints.

## Challenges

1. **Massive Review Churn:** Story 3.1 experienced 25 AI review rounds and Story 3.3 experienced 14 rounds. This drastically reduced velocity.
2. **Repeating Past Mistakes:** Action items from Epic 2 (like adding "in-flight guards" for UI mutations and front-loading negative test scenarios) were not systemically applied to the AI workflows. The AI agents simply "forgot" to follow the retro advice, leading to the same bugs being re-introduced and caught during review.
3. **Complex State Management:** Integrating drag-and-drop (`@dnd-kit`) with React state and mock API promises resulted in tricky visual flashes and difficult-to-test async behavior (requiring multiple rounds to fix).
4. **N+1 Lazy Load Regressions:** Both backend stories repeatedly encountered N+1 SQLAlchemy lazy-loading regressions upon introducing new nested relationships.

## Key Insights

1. **Process Must Be Mechanized:** Writing action items in a markdown retrospective document is useless for AI agents unless those insights are explicitly piped into their instructions (e.g., updating the `create-story` workflow or agent configs directly).
2. **Eager Loading is Hard:** SQLAlchemy's `expire_on_commit` combined with FastAPI's response serialization frequently causes implicit N+1 queries. We need a systematic check for `selectinload` on all nested resources.
